# LightDo 项目能力分析

## 1. 结论摘要

当前仓库更接近一个 **Electron + React 桌面待办应用的项目骨架**，而不是一份可直接证明全部产品能力的完整实现。

从现有文件可以确认：

- 这是一个面向 **Windows** 的桌面应用项目。
- 技术路线是 **Electron 主进程 + React 渲染进程 + TypeScript + Vite**。
- 产品形态明确包含 **主窗口** 和 **悬浮球窗口** 两个前端入口。
- 已具备 **开发、构建、打包、便携版发布、安装版发布** 的工程能力设计。
- 打包目标明确支持 **Windows portable** 和 **NSIS 安装包**。

但同时也必须说明：

- 仓库内 **缺少 `src/` 源码目录**，无法验证核心业务是否真实实现。
- `README.md` 中列出的多数功能，目前只能视为 **产品设计声明**，不能视为仓库已实现能力。
- HTML、TypeScript 配置和打包配置均引用 `src/` 下文件，说明项目原本应存在完整源码，但当前仓库快照不完整。

## 2. 分析依据

本分析基于当前仓库内实际存在的文件：

- `README.md`
- `package.json`
- `vite.config.ts`
- `electron-builder.yml`
- `tsconfig.json`
- `tsconfig.main.json`
- `index.html`
- `floating-ball.html`

未发现以下关键内容：

- `src/main/**`
- `src/renderer/**`
- `src/shared/**`
- `README_EN.md`
- `RTK.md`

其中，`AGENTS.md` 指向 `@RTK.md`，但当前仓库内未找到该文件，因此无法纳入约束分析。

## 3. 已验证能力

这一部分只列出能够被现有仓库文件直接证明的能力。

### 3.1 桌面应用基础能力

项目具备标准 Electron 桌面应用结构设计：

- `package.json` 的 `main` 指向 `dist/main/main/index.js`
- 存在主窗口入口 `index.html`
- 存在悬浮球窗口入口 `floating-ball.html`
- `tsconfig.main.json` 与 `tsconfig.json` 分离，说明主进程与渲染进程独立构建

可确认该项目的目标产品不是网页待办，而是本地桌面程序。

### 3.2 双窗口产品形态

从两个 HTML 入口可以确认项目设计为双窗口形态：

- `index.html` 加载 `/src/renderer/main.tsx`
- `floating-ball.html` 加载 `/src/renderer/floating-ball.tsx`

这意味着项目至少在架构层面支持：

- 主待办窗口
- 独立悬浮球窗口
- 两个渲染入口的分别构建与运行

### 3.3 前端技术能力

从依赖和构建配置可以确认以下前端能力栈：

- React 18
- TypeScript 5
- Vite 5
- `@vitejs/plugin-react`

说明项目具备：

- React 组件化开发能力
- TypeScript 类型化开发能力
- Vite 本地开发与生产构建能力

### 3.4 拖拽排序能力的技术预留

`package.json` 依赖包含：

- `@dnd-kit/core`
- `@dnd-kit/sortable`
- `@dnd-kit/utilities`

这可以确认项目对“拖拽排序”有明确技术选型，说明：

- 产品设计中存在列表拖拽排序场景
- 工程上已为拖拽交互预置依赖

但由于源码缺失，无法确认拖拽逻辑是否已真正实现。

### 3.5 开发与调试能力

从 `scripts` 可以确认项目具备完整开发链路：

- `npm run dev`
- `npm run watch:main`
- `npm run build:main`
- `npm run electron`
- `npm run build`
- `npm run type-check`

这说明工程层面已设计：

- 主进程 TypeScript 编译
- Vite 开发服务器
- Electron 联调启动
- 类型检查
- 生产构建

`dev` 脚本的流程为：

1. 先构建主进程
2. 并行监听主进程变更
3. 启动 Vite
4. 等待前端服务后再启动 Electron

这是一套标准且可维护的 Electron 开发流程设计。

### 3.6 Windows 发布能力

从 `electron-builder.yml` 可以确认项目已配置 Windows 分发：

- `portable` 便携版
- `nsis` 安装版
- 输出目录为 `release/`
- 支持桌面快捷方式和开始菜单快捷方式
- 支持用户选择安装目录

因此可以确认项目具备：

- 生成免安装可执行文件的能力
- 生成安装程序的能力
- 面向 Windows 终端用户发布的能力

### 3.7 本地部署友好能力

从 README 与打包配置可确认项目定位偏向终端可执行分发，而非要求用户安装开发环境。这意味着项目在产品定位上具备：

- 本地离线运行
- 便携式分发
- 面向普通 Windows 用户交付

这属于产品交付能力，而非业务功能能力。

## 5. 按能力域拆解项目全貌

如果从“项目应该具备什么能力”角度来拆，这个仓库当前可以归纳为以下几个能力域。

### 5.1 产品能力域

从 README 设计目标可推断项目 intended capabilities 包括：

- 个人待办事项管理
- 极简快速录入
- 桌面悬浮入口
- 低打扰后台运行
- 快速唤起与快速隐藏

这反映的是产品定位，不等于当前仓库可验证实现。

### 5.2 桌面端架构能力域

当前可明确看出项目采用：

- Electron 主进程负责系统级能力
- React 渲染进程负责界面
- 双页面入口支撑主窗口与悬浮球窗口
- TypeScript 统一主进程和前端类型体系

这部分是当前仓库最清晰、最可信的能力。

### 5.3 工程化能力域

项目具备较完整的工程化基础设施设计：

- 本地开发脚本
- 类型检查
- 主进程独立编译
- 渲染进程独立构建
- Electron 联调
- Windows 打包发布

如果补齐源码，这套工程骨架可以直接承载正式开发。

### 5.4 发布与交付能力域

项目对终端用户交付较明确：

- 便携版分发
- 安装版分发
- 桌面快捷方式
- 开始菜单入口
- Windows 桌面使用场景

这意味着项目不仅是 demo 架构，而是有明确的产品交付目标。

## 6. 当前缺口与风险

### 6.1 最大缺口：核心源码缺失

当前最关键的问题不是功能多少，而是：

- `src/` 目录整体缺失
- 构建入口引用的源码不存在
- README 描述无法被代码证明

这直接导致：

- 无法运行
- 无法构建
- 无法测试
- 无法确认 README 中的功能真伪和完成度

### 6.2 文档与仓库状态不一致

`README.md` 描述的是一份相对完整的应用，但仓库实际只有配置与入口壳文件。这种不一致会带来：

- 接手者误判项目成熟度
- 无法根据 README 快速启动
- 功能评估和开发排期失真

### 6.3 约束文件缺失

仓库说明提到 `@RTK.md`，但实际缺失。这说明：

- 项目协作规范可能未随仓库同步
- 自动化代理或接手开发者拿不到完整工作约束

## 7. 建议的能力分级结论

为了后续沟通准确，建议将本项目能力按以下分级表达。

### A 级：已确认存在

- Electron 桌面应用工程骨架
- React + TypeScript + Vite 前端方案
- 主进程 / 渲染进程分离构建
- 主窗口 + 悬浮球双入口
- Windows 便携版与安装版打包配置
- 本地开发、构建、类型检查脚本
- 拖拽排序依赖选型

### B 级：高度可能存在于原始设计，但当前仓库无法证明

- 待办增删改查
- 完成状态切换
- 拖拽排序逻辑
- 悬浮球拖动与显隐主窗口
- 全局快捷键
- 本地 JSON 存储
- 设置面板
- 开机自启
- 点击外部自动隐藏
- 窗口智能定位
- 毛玻璃 UI

### C 级：当前仓库明确不具备

- 可执行的完整源码交付
- 可直接运行的业务实现
- 可验证测试
- 可复现实机功能演示

## 8. 一句话总结

LightDo 当前仓库可被定义为：

**一个面向 Windows 的 Electron 桌面待办应用骨架，具备双窗口架构、React/Vite 工程体系和发布配置，但缺失核心源码，因此除工程与发布层能力外，大部分产品功能目前只能视为文档声明，不能认定为已实现能力。**
